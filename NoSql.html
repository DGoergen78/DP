Was ist NoSQL?

NoSQL steht für „Not only SQL" – Datenbanksysteme, die über das klassische relationale Modell hinausgehen und flexiblere Datenstrukturen ermöglichen.

Das Problem mit klassischen Datenbanken

Relationale Datenbanken (SQL) speichern Daten in starren Tabellen mit festen Spalten. Das funktioniert gut – bis die Anforderungen wachsen:

Problem

Skalierung

SQL-Datenbanken skalieren primär vertikal: mehr RAM, stärkere CPU. Das ist teuer und hat physikalische Grenzen – irgendwann gibt es keinen leistungsstärkeren Server mehr zu kaufen.

Warum können NoSQL-DBs einfach horizontal skalieren?
Weil ein NoSQL-Dokument alle nötigen Daten in sich trägt – es hat keine Fremdschlüsselbeziehungen zu anderen Dokumenten auf anderen Servern. Man kann ein Dokument daher auf einem beliebigen Server speichern. Bei SQL hingegen müsste ein JOIN Daten von mehreren Servern zusammenführen – das erfordert aufwändige Netzwerkkommunikation. NoSQL verteilt einfach ganze Dokumente auf viele Server und skaliert damit quasi beliebig.
Problem

Starre Struktur

Jede Zeile muss dieselben Spalten haben. Unterschiedliche Produkte (Bücher, Schuhe, Elektronik) haben aber völlig andere Eigenschaften – in SQL entstehen viele leere NULL-Felder.

Beispiel: Ein Buch hat ISBN und Seitenzahl, ein T-Shirt hat Größe und Farbe. In einer gemeinsamen SQL-Tabelle hätte ein Buch die Felder „Größe" und „Farbe" leer (NULL), ein T-Shirt hätte „ISBN" und „Seitenzahl" leer. In NoSQL hat jedes Dokument genau die Felder, die es braucht – keine anderen.
Problem

Verschachtelte Daten & JOINs

Ein Nutzer mit mehreren Adressen braucht in SQL zwei separate Tabellen und einen JOIN. Je komplexer die Daten, desto mehr JOINs – und JOINs werden bei großen Tabellen sehr langsam.

Warum braucht man überhaupt JOINs?
SQL speichert zusammengehörige Daten bewusst in getrennten Tabellen (Normalisierung), um Redundanz zu vermeiden. Möchte man die Daten zusammen abfragen, muss die DB jede Zeile aus Tabelle A mit jeder passenden Zeile aus Tabelle B abgleichen – das ist der JOIN. Bei Millionen Einträgen dauert das lang und belastet den Server stark. NoSQL speichert alles in einem Dokument – ein einziger Lesezugriff genügt.
Problem

Schema-Änderungen

Eine neue Spalte hinzufügen erfordert ein ALTER TABLE – bei großen Tabellen aufwendig und riskant.

Warum ist ALTER TABLE riskant?
Die Datenbank muss dabei jede einzelne Zeile der Tabelle anfassen und das neue Feld ergänzen. Bei 100 Millionen Zeilen dauert das Stunden – in dieser Zeit ist die Tabelle oft gesperrt und für Anwendungen nicht nutzbar. Schlägt die Operation fehl, muss alles zurückgerollt werden. In NoSQL fügt man neue Felder einfach in einzelne Dokumente ein, ohne die restlichen Dokumente zu berühren.

NoSQL als Lösung

Die populärsten NoSQL-Datenbanken sind dokumentenorientiert (z.B. MongoDB). Daten werden als JSON-ähnliche Dokumente gespeichert – flexibel, verschachtelbar und ohne festes Schema.

NoSQL — alles in einem Dokument
{
  "_id": "nutzer_001",
  "name": "Anna Müller",
  "email": "[email protected]",
  "adressen": [
    { "typ": "privat", "stadt": "Berlin" },
    { "typ": "arbeit",  "stadt": "Hamburg" }
  ],
  "bestellungen": 42
}
SQL — 2 Tabellen, JOIN nötig
-- Tabelle 1
CREATE TABLE nutzer (
  id    INT PRIMARY KEY,
  name  VARCHAR(100),
  email VARCHAR(200)
);
-- Tabelle 2
CREATE TABLE adressen (
  id        INT PRIMARY KEY,
  nutzer_id INT REFERENCES nutzer,
  typ       VARCHAR(50),
  stadt     VARCHAR(100)
);
-- Abfrage braucht JOIN:
SELECT n.name, a.stadt
FROM nutzer n
JOIN adressen a ON a.nutzer_id = n.id;

Im NoSQL-Beispiel sind alle Daten einer Person in einem einzigen Dokument gespeichert – kein JOIN nötig. In SQL müssen beide Tabellen explizit per JOIN zusammengeführt werden.

Wann setzt man NoSQL ein?

Einsatzgebiet

Big Data & hohe Last

Social Media, IoT-Sensordaten, Logs – Millionen Schreibvorgänge pro Sekunde, horizontal auf vielen Servern verteilt.

Einsatzgebiet

Flexible Schemas

Produkte mit sehr unterschiedlichen Attributen in einer Collection – ohne NULL-Felder und ohne ALTER TABLE.

Einsatzgebiet

Verschachtelte Strukturen

Blog-Posts, Nutzerprofile, Konfigurationen – Daten, die natürlicherweise als verschachteltes JSON vorliegen.

Einsatzgebiet

Schnelle Entwicklung

Kein Schema-Design vorab nötig. Felder können jederzeit ergänzt werden. Ideal für agile Projekte und Prototypen.

ACID vs BASE

SQL

ACID

Atomicity – Alles oder nichts: eine Transaktion wird vollständig oder gar nicht ausgeführt.
Consistency – Die DB ist immer in einem gültigen Zustand.
Isolation – Parallele Transaktionen stören sich gegenseitig nicht.
Durability – Einmal gespeichert bleibt gespeichert, auch bei Absturz.

Ideal für Banktransaktionen, wo jeder Cent stimmen muss.

NoSQL

BASE

Basically Available – Das System antwortet immer, auch wenn nicht alle Daten aktuell sind.
Soft state – Der Zustand der Daten darf sich noch ändern.
Eventually consistent – Alle Knoten werden irgendwann denselben Stand haben.

Ideal für Social Media: Ein Like darf kurz verzögert erscheinen – aber das System soll nie komplett ausfallen.

JSON als Datenformat

Dokumentenorientierte Datenbanken speichern Daten im JSON-Format (JavaScript Object Notation). JSON besteht aus Schlüssel-Wert-Paaren in geschweiften Klammern. Der Typ eines Wertes bestimmt, wie er geschrieben wird – das ist besonders beim Umwandeln von Tabellendaten in JSON-Dokumente prüfungsrelevant:

String (Text)
"Berlin"
Text – immer in doppelten Anführungszeichen
Number (Zahl)
42  /  3.14
Zahl – ohne Anführungszeichen
Boolean
true  /  false
Wahrheitswert – klein geschrieben, ohne Anführungszeichen
null
null
Kein Wert / unbekannt
Array (Liste)
["Berlin", "Hamburg"]
Mehrere Werte – in eckigen Klammern
Object (Objekt)
{"stadt": "Berlin"}
Verschachteltes Dokument in geschweiften Klammern
